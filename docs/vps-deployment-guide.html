☁️ Guia de Deploy — Baluarte em VPS (Contabo)

Migração do Railway + Vercel para uma VPS Contabo VPS S (4 vCPU, 8GB RAM, 75GB NVMe, €4,50/mês).
Inclui: setup inicial, domínio, CI/CD, banco, backups, monitoramento e precificação.

2. Setup Inicial da VPS

2.1 Acessar a VPS

Após contratar a Contabo, você recebe o IP e senha root por e-mail.

ssh root@<ip-da-vps>

2.2 Primeiros passos

apt update && apt upgrade -y
apt install -y docker.io docker-compose-v2 nginx certbot \
  python3-certbot-nginx ufw htop neofetch postgresql-client

# Cria usuário não-root para segurança
adduser deploy
usermod -aG docker deploy
usermod -aG sudo deploy

2.3 Firewall (UFW)

ufw default deny incoming
ufw default allow outgoing
ufw allow ssh
ufw allow 80/tcp
ufw allow 443/tcp
ufw --force enable
ufw status verbose
🔒 Portas: Apenas 22 (SSH), 80 (HTTP) e 443 (HTTPS) abertas. O Docker expõe internamente 8080, 3000, 5432, etc., mas o nginx faz o proxy reverso e só libera 80/443.

2.4 SSH Key (acesso sem senha)

# No seu computador local:
ssh-copy-id deploy@<ip-da-vps>
# Ou copie manualmente a chave pública para ~/.ssh/authorized_keys

# Na VPS, desabilite login por senha:
sudo sed -i 's/PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl restart sshd

3. Domínio e SSL

3.1 Apontar o DNS

No painel do seu provedor de domínio, crie os seguintes registros:

TipoNomeValor
A@<ip-da-vps>
Aapi<ip-da-vps>
Awww<ip-da-vps>

Se usar Cloudflare, coloque o DNS proxy como DNS only (nuvem cinza) até o SSL estar configurado, depois mude para Proxied (nuvem laranja).

3.2 Configurar nginx

sudo nano /etc/nginx/sites-available/baluarte
server {
    listen 80;
    server_name seudominio.com.br api.seudominio.com.br;

    # Frontend Next.js
    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }

    # API Backend
    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Uploads
    location /uploads/ {
        alias /var/www/uploads/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }
}
sudo ln -s /etc/nginx/sites-available/baluarte /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx

3.3 SSL com Certbot

sudo certbot --nginx -d seudominio.com.br -d www.seudominio.com.br
sudo certbot --nginx -d api.seudominio.com.br

# Verifica renovação automática
sudo certbot renew --dry-run
🔁 Renovação automática: O Certbot cria um systemd timer que renova automaticamente. Para verificar: systemctl list-timers | grep certbot.

5. CI/CD com GitHub Actions

Dois workflows separados: um para o backend (Spring Boot native) e um para o frontend (Next.js).

5.1 Backend — Build + Deploy

Crie em .github/workflows/backend-deploy.yml:

name: Backend Deploy

on:
  push:
    branches: [main]
    paths:
      - 'Baluarte-core/**'
      - '.github/workflows/backend-deploy.yml'

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ghcr.io/${{ github.repository }}/baluarte-core-native

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4

      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build native image
        run: |
          docker build \
            -t ${{ env.IMAGE_NAME }}:${{ github.sha }} \
            -t ${{ env.IMAGE_NAME }}:latest \
            ./Baluarte-core

      - name: Push image
        run: |
          docker push ${{ env.IMAGE_NAME }}:${{ github.sha }}
          docker push ${{ env.IMAGE_NAME }}:latest

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Deploy na VPS via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          script: |
            cd /home/deploy/baluarte
            docker compose pull core
            docker compose up -d --no-deps core
            docker image prune -f

6. Banco de Dados — Backup e Prevenção

6.1 Backup automático diário

Crie o script /home/deploy/baluarte/backup.sh:

nano /home/deploy/baluarte/backup.sh
#!/bin/bash
set -e

BACKUP_DIR="/home/deploy/baluarte/backups"
DB_NAME="baluarte"
DB_USER="baluarte"
DB_PASS="$(grep POSTGRES_PASSWORD /home/deploy/baluarte/.env | cut -d= -f2)"
DATE="$(date +%Y-%m-%d_%H-%M)"
RETENTION_DAYS=30
RCLONE_REMOTE="cloudflare:baluarte-backups"

mkdir -p "$BACKUP_DIR"

# Dump via container (conexão interna, sem expor porta)
docker exec baluarte-db pg_dump -U "$DB_USER" "$DB_NAME" \
  --no-owner --no-privileges \
  | gzip > "$BACKUP_DIR/baluarte_$DATE.sql.gz"

# Remove backups antigos locais
find "$BACKUP_DIR" -name "baluarte_*.sql.gz" -mtime +$RETENTION_DAYS -delete

# Envia para Cloudflare R2 (ou S3 compatível)
rclone copy "$BACKUP_DIR/baluarte_$DATE.sql.gz" "$RCLONE_REMOTE" --progress

echo "Backup concluído: baluarte_$DATE.sql.gz"
chmod +x /home/deploy/baluarte/backup.sh

6.2 Agendar no crontab

crontab -e
# Backup do banco todos os dias às 03:00
0 3 * * * /home/deploy/baluarte/backup.sh >> /var/log/baluarte-backup.log 2>&1

# Backup semanal completo (segunda 04:00)
0 4 * * 1 /home/deploy/baluarte/backup.sh --full >> /var/log/baluarte-backup-full.log 2>&1

6.3 Testar restore

# Restaurar de um backup específico
gunzip -c backups/baluarte_2026-06-03_03-00.sql.gz | docker exec -i baluarte-db psql -U baluarte -d baluarte
☢️ Teste o restore periodicamente! Um backup que não pode ser restaurado não é um backup. Faça um restore em banco de teste a cada 30 dias.

6.4 Cloudflare R2 (ou similar)

Armazenamento externo barato para backups:

sudo apt install rclone

Estratégia de backup: Diário (retém 30 dias) + semanal (retém 6 meses) + mensal (retém 1 ano). O script acima faz diário. Adapte para sua necessidade.

6.5 Prevenção de perda de dados

  • Snapshot da VPS: A Contabo oferece 1 snapshot incluso. Agende semanal via painel.
  • RPO (Recovery Point Objective): Máximo 24 horas (backup diário).
  • RTO (Recovery Time Objective): ~15 minutos (restore do último dump + docker compose up).
  • Offsite: Backup no Cloudflare R2 (ou Backblaze) para proteger contra falha da VPS.
  • Migração segura: Sempre faça dump do banco atual antes de aplicar migrations destrutivas.

7.3 Health check simples (cronjob)

nano /home/deploy/baluarte/healthcheck.sh
#!/bin/bash
WEBHOOK_TELEGRAM="https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/sendMessage"
CHAT_ID="$TELEGRAM_CHAT_ID"
SERVICES=(
  "https://seudominio.com.br"
  "https://api.seudominio.com.br/actuator/health"
)

for URL in "${SERVICES[@]}"; do
  STATUS=$(curl -o /dev/null -s -w "%{http_code}" "$URL")
  if [ "$STATUS" != "200" ] && [ "$STATUS" != "302" ]; then
    curl -s "$WEBHOOK_TELEGRAM" \
      -d "chat_id=$CHAT_ID" \
      -d "text=🔴 Baluarte OFFLINE: $URL retornou $STATUS"
  fi
done
# No crontab, a cada 5 minutos
*/5 * * * * /home/deploy/baluarte/healthcheck.sh

FROM node:22-alpine AS base

FROM base AS deps
WORKDIR /app
COPY package.json package-lock.json ./

FROM base AS builder
WORKDIR /app
COPY package.json package-lock.json ./

COPY . .

FROM base AS runner

10. Checklist de Migração

  • Contratar Contabo VPS S
  • Instalar Docker, nginx, certbot, ufw na VPS
  • Configurar firewall (apenas 22, 80, 443)
  • Criar usuário deploy e configurar SSH key
  • Apontar domínio (A records) para o IP da VPS
  • Configurar nginx + SSL (certbot)
  • Exportar banco do Railway (pg_dump)
  • Importar banco na VPS (psql)
  • Subir docker compose com .env real
  • Validar health check: /actuator/health
  • Validar frontend: carregar home, login, busca, produto
  • Configurar webhooks Mercado Pago / SuperFrete para novo domínio
  • Configurar GitHub Actions secrets (VPS_HOST, VPS_USER, VPS_SSH_KEY)
  • Fazer push na main e validar CI/CD
  • Agendar backup automático (crontab + rclone)
  • Configurar monitoramento (Uptime Kuma ou cron + Telegram)
  • Atualizar sitemap, OG metadata e robots.txt com domínio novo
  • Desativar Railway (apenas depois de tudo validado)
