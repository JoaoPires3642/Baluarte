Guia de Deploy — Baluarte em VPS (Contabo)

Arquitetura híbrida: Frontend na Vercel (grátis) + Backend/banco na VPS Contabo (€4,50 fixo) + Builds no GitHub Actions.
Migração do Railway para VPS com custo fixo e mais recursos.

3. Setup Inicial da VPS

3.1 Acessar a VPS

ssh root@<ip-da-vps>

3.2 Primeiros passos

apt update && apt upgrade -y
apt install -y docker.io docker-compose-v2 nginx certbot \
  python3-certbot-nginx ufw htop neofetch postgresql-client

# Cria usuário não-root para segurança
adduser deploy
usermod -aG docker deploy
usermod -aG sudo deploy

3.3 Firewall (UFW)

ufw default deny incoming
ufw default allow outgoing
ufw allow ssh
ufw allow 80/tcp
ufw allow 443/tcp
ufw --force enable
ufw status verbose
🔒 Portas: Apenas 22 (SSH), 80 (HTTP) e 443 (HTTPS) abertas. O Docker expõe o backend internamente na 8080. Tudo passa pelo nginx.

3.4 SSH Key (acesso sem senha)

# No seu computador local:
ssh-copy-id deploy@<ip-da-vps>

# Na VPS, desabilite login por senha:
sudo sed -i 's/PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
sudo systemctl restart sshd

4. Domínio e SSL

4.1 Apontar o DNS

TipoNomeValor
Aapi<ip-da-vps>

O frontend fica na Vercel (domínio seudominio.com.br). Apenas o subdomínio api.seudominio.com.br aponta para a VPS.

4.2 Configurar nginx

sudo nano /etc/nginx/sites-available/baluarte-api
server {
    listen 80;
    server_name api.seudominio.com.br;

    # API Backend
    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Uploads (para servir imagens diretamente)
    location /uploads/ {
        alias /var/www/uploads/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }
}
sudo ln -s /etc/nginx/sites-available/baluarte-api /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx

4.3 SSL com Certbot

sudo certbot --nginx -d api.seudominio.com.br
sudo certbot renew --dry-run
🔁 Renovação automática: O Certbot cria um systemd timer que renova automaticamente.

5. Docker Compose — Produção

A VPS roda apenas backend + banco. O frontend fica na Vercel.

5.1 Estrutura na VPS

5.2 docker-compose.yml

services:
  db:
    image: postgres:16-alpine
    container_name: baluarte-db
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: baluarte
    ports:
      - "127.0.0.1:5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d baluarte"]
      interval: 10s
      timeout: 5s
      retries: 5

  core:
    image: ghcr.io/joaopires3642/baluarte-core-native:latest
    container_name: baluarte-core
    restart: unless-stopped
    pull_policy: always
    env_file: .env
    depends_on:
      db: { condition: service_healthy }
    ports:
      - "127.0.0.1:8080:8080"
    volumes:
      - uploads_data:/var/www/uploads

volumes:
  postgres_data:
  uploads_data:
⚠️ Segurança: 127.0.0.1:5432:5432 impede acesso externo direto ao PostgreSQL. Só o backend conecta. As portas do backend e banco NÃO ficam expostas publicamente — apenas o nginx na porta 443 enxerga o backend.

6. CI/CD com GitHub Actions

O Build das imagens acontece no GitHub Actions. O deploy do frontend é automático via Vercel (conectado ao GitHub).

⏱️ Tempo de build nativo (GraalVM): ~10 minutos por build.
⚠️ Atenção: O build nativo do backend (GraalVM) é pesado. Enquanto roda, consome ~8GB de RAM no runner do GitHub. Se o workflow falhar com "out of memory", troque para o Dockerfile JVM (build em ~3 min, consome menos RAM, mas o container usa mais RAM em runtime).

6.1 Limites do GitHub Actions Free

Minutos/mês (repositórios privados)2.000 min
Storage de artefatos500 MB

O repositório Baluarte é público, então os builds são gratuitos e ilimitados. Não há contagem de minutos.

6.2 Backend — Build + Deploy

Arquivo .github/workflows/backend-deploy.yml:

name: Backend Deploy

on:
  push:
    branches: [main]
    paths:
      - 'Baluarte-core/**'
      - '.github/workflows/backend-deploy.yml'

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ghcr.io/${{ github.repository }}/baluarte-core-native

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4

      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build native image
        run: |
          docker build \
            -t ${{ env.IMAGE_NAME }}:${{ github.sha }} \
            -t ${{ env.IMAGE_NAME }}:latest \
            ./Baluarte-core

      - name: Push image
        run: |
          docker push ${{ env.IMAGE_NAME }}:${{ github.sha }}
          docker push ${{ env.IMAGE_NAME }}:latest

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Deploy na VPS via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          script: |
            cd /home/deploy/baluarte
            docker compose pull core
            docker compose up -d --no-deps core
            docker image prune -f

7. Banco de Dados — Backup e Prevenção

7.1 Backup automático diário

nano /home/deploy/baluarte/backup.sh
#!/bin/bash
set -e

BACKUP_DIR="/home/deploy/baluarte/backups"
DB_NAME="baluarte"
DB_USER="baluarte"
DB_PASS="$(grep POSTGRES_PASSWORD /home/deploy/baluarte/.env | cut -d= -f2)"
DATE="$(date +%Y-%m-%d_%H-%M)"
RETENTION_DAYS=30
RCLONE_REMOTE="cloudflare:baluarte-backups"

mkdir -p "$BACKUP_DIR"

docker exec baluarte-db pg_dump -U "$DB_USER" "$DB_NAME" \
  --no-owner --no-privileges \
  | gzip > "$BACKUP_DIR/baluarte_$DATE.sql.gz"

find "$BACKUP_DIR" -name "baluarte_*.sql.gz" -mtime +$RETENTION_DAYS -delete

rclone copy "$BACKUP_DIR/baluarte_$DATE.sql.gz" "$RCLONE_REMOTE" --progress

echo "Backup concluído: baluarte_$DATE.sql.gz"
chmod +x /home/deploy/baluarte/backup.sh

7.2 Agendar no crontab

crontab -e
# Backup diário às 03:00
0 3 * * * /home/deploy/baluarte/backup.sh >> /var/log/baluarte-backup.log 2>&1

7.3 Testar restore

gunzip -c backups/baluarte_2026-06-03_03-00.sql.gz | docker exec -i baluarte-db psql -U baluarte -d baluarte
Teste o restore periodicamente! Um backup que não pode ser restaurado não é um backup. Faça um restore em banco de teste a cada 30 dias.

7.5 Prevenção de perda de dados

  • Snapshot Contabo: 1 incluso. Agende semanal via painel.
  • RPO: Máximo 24 horas (backup diário).
  • RTO: ~15 minutos (restore + docker compose up).
  • Offsite: Backup no Cloudflare R2 (ou Backblaze).
  • Migração segura: Sempre faça dump antes de migrations destrutivas.

8.2 Health check com cron + Telegram

nano /home/deploy/baluarte/healthcheck.sh
#!/bin/bash
WEBHOOK_TELEGRAM="https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/sendMessage"
CHAT_ID="$TELEGRAM_CHAT_ID"
SERVICES=(
  "https://api.seudominio.com.br/actuator/health"
)

for URL in "${SERVICES[@]}"; do
  STATUS=$(curl -o /dev/null -s -w "%{http_code}" "$URL")
  if [ "$STATUS" != "200" ]; then
    curl -s "$WEBHOOK_TELEGRAM" \
      -d "chat_id=$CHAT_ID" \
      -d "text=Baluarte OFFLINE: $URL retornou $STATUS"
  fi
done
chmod +x /home/deploy/baluarte/healthcheck.sh
crontab -e
# A cada 5 minutos
*/5 * * * * /home/deploy/baluarte/healthcheck.sh

9. Dockerfile do Backend

O Dockerfile do backend (Baluarte-core/Dockerfile) já existe no repositório. Duas opções:

Native (GraalVM)JVM (padrão)
Build time~10 min~3 min
RAM em runtime~60 MB~200-300 MB
Startup<0.5s~10s
Indicado paraRecurso apertadoVPS com 8GB de RAM
Recomendação para a VPS: Use a imagem nativa se o GitHub Actions conseguir buildar. A economia de RAM (60MB vs 300MB) é irrelevante com 8GB, mas o startup quase instantâneo é bom para restart rápido.

Se o build nativo falhar por RAM no GitHub Actions, crie um Dockerfile JVM alternativo:

FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -DskipTests

FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
RUN addgroup -S baluarte && adduser -S baluarte -G baluarte
COPY --from=builder /app/target/*.jar app.jar
USER baluarte
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

11. Checklist de Migração

  • Contratar Contabo VPS S
  • Instalar Docker, nginx, certbot, ufw na VPS
  • Configurar firewall (apenas 22, 80, 443)
  • Criar usuário deploy e configurar SSH key
  • Apontar api.seudominio.com.br (A record) para o IP da VPS
  • Configurar nginx + SSL para api.seudominio.com.br
  • Exportar banco do Railway (pg_dump)
  • Importar banco na VPS (psql)
  • Subir docker compose com .env real
  • Validar health check: /actuator/health
  • Atualizar NEXT_PUBLIC_API_BASE_URL na Vercel
  • Validar frontend: home, login, busca, produto
  • Configurar webhooks Mercado Pago / SuperFrete para api.seudominio.com.br
  • Configurar GitHub Actions secrets (VPS_HOST, VPS_USER, VPS_SSH_KEY)
  • Fazer push na main e validar CI/CD
  • Agendar backup automático (crontab + rclone)
  • Configurar monitoramento (cron + Telegram)
  • Atualizar sitemap, OG metadata e robots.txt com o domínio novo
  • Desativar Railway (apenas depois de tudo validado)
