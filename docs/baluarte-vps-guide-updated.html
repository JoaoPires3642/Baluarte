Vantagens

  • Preço Imbatível: Impossível achar 8GB de RAM por ~R$20 em AWS, Google ou Azure.
  • Tráfego Ilimitado: Sem surpresas na conta no fim do mês.
  • Snapshots: Permite "congelar" o estado do servidor antes de uma mudança arriscada.
⚠️ O Suporte é em Inglês
A Contabo é excelente, mas o painel e o suporte são em inglês/alemão. O suporte técnico direto demora cerca de 24h para responder, por isso o monitoramento local é essencial.
Negócio

O que o Cliente Recebe

Diferenciais de vender uma solução em VPS própria em vez de "hospedagem comum".

💎 Estabilidade Premium

O site nunca "dorme". A resposta da API é instantânea porque os recursos são dedicados apenas ao projeto dele.

🛡️ Segurança de Dados

Backups automáticos em local diferente (Cloudflare R2). Se a VPS explodir, os dados estão salvos em outro continente.

⚡ Performance

Imagens e arquivos carregam rápido. O banco de dados PostgreSQL 16 otimizado garante buscas velozes em produtos e pedidos.

Domínio & SSL

Configuração do Nginx como Proxy Reverso e Certbot para HTTPS.

DNS

Aponte um registro A no seu provedor de domínio:

api.seudominio.com.br  ->  IP_DA_VPS

Nginx Config

nano /etc/nginx/sites-available/baluarte
server {
    server_name api.seudominio.com.br;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    location /uploads/ {
        alias /home/deploy/baluarte/data/uploads/;
        expires 30d;
    }
}
ln -s /etc/nginx/sites-available/baluarte /etc/nginx/sites-enabled/
nginx -t && systemctl reload nginx

SSL Grátis

apt install certbot python3-certbot-nginx -y
certbot --nginx -d api.seudominio.com.br

Deploy

CI/CD — GitHub Actions

Push na main → build nativo → push GHCR → deploy SSH na VPS.

Workflow completo

name: Build & Deploy

on:
  push:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ghcr.io/joaopires3642/baluarte-core-native

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - name: Build native image
        run: |
          docker build \
            -t ${{ env.IMAGE_NAME }}:${{ github.sha }} \
            -t ${{ env.IMAGE_NAME }}:latest \
            ./Baluarte-core
      - name: Push image
        run: |
          docker push ${{ env.IMAGE_NAME }}:${{ github.sha }}
          docker push ${{ env.IMAGE_NAME }}:latest

  deploy:
    needs: build
    runs-on: ubuntu-latest
    steps:
      - name: Deploy na VPS via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          script: |
            cd /home/deploy/baluarte
            docker compose pull core
            docker compose up -d --no-deps core
            docker image prune -f

Frontend (Vercel)

Zero configuração. A Vercel já está conectada ao GitHub — push na main dispara o build automaticamente. Variável que precisa estar correta:

NEXT_PUBLIC_API_BASE_URL=https://api.seudominio.com.br/api/v1
💡 Se o build nativo estoura RAM
O build GraalVM consome ~8 GB de RAM no runner. Se falhar com "out of memory", troque para o Dockerfile JVM (build em ~3 min, mais simples). Com 8 GB de RAM na VPS, os 300 MB de runtime do JVM são irrelevantes.

Dockerfile do Backend

GraalVM Native (startup <0.5s, 60 MB RAM) vs JVM padrão (build mais simples, ~300 MB RAM).

Comparação

CaracterísticaNative (GraalVM)JVM (padrão)
Tempo de build~10 min~3 min
RAM em runtime~60 MB~200–300 MB
Startup<0.5 s~10 s
RiscoReflexão/proxies Spring podem precisar de hintsSem surpresas
⚠️ GraalVM + Mercado Pago
Libs de pagamento às vezes precisam de hints de reflexão em native-image.properties. Se compilar mas falhar em runtime com ClassNotFoundException, use o Dockerfile JVM — com 8 GB na VPS, os 300 MB são irrelevantes.

  "-XX:+UseContainerSupport",
  "-XX:MaxRAMPercentage=75.0",

Backup & Restore

RPO: 24h · RTO: ~15 min · Offsite automático para Cloudflare R2

Script de backup

nano /home/deploy/baluarte/backup.sh
#!/bin/bash
set -e

BACKUP_DIR="/home/deploy/baluarte/backups"
DB_NAME="baluarte"
DB_USER="baluarte"
# -f2- captura senhas com "=" dentro (ex: base64)
DB_PASS="$(grep ^POSTGRES_PASSWORD /home/deploy/baluarte/.env | cut -d= -f2-)"
DATE="$(date +%Y-%m-%d_%H-%M)"
RETENTION_DAYS=30
RCLONE_REMOTE="cloudflare:baluarte-backups"

mkdir -p "$BACKUP_DIR"

docker exec baluarte-db pg_dump -U "$DB_USER" "$DB_NAME" \
  --no-owner --no-privileges \
  | gzip > "$BACKUP_DIR/baluarte_$DATE.sql.gz"

find "$BACKUP_DIR" -name "baluarte_*.sql.gz" -mtime +$RETENTION_DAYS -delete

rclone copy "$BACKUP_DIR/baluarte_$DATE.sql.gz" "$RCLONE_REMOTE" --progress

echo "Backup concluído: baluarte_$DATE.sql.gz"
chmod +x /home/deploy/baluarte/backup.sh

crontab -e
# Backup diário às 03h00
0 3 * * * /home/deploy/baluarte/backup.sh >> /var/log/baluarte-backup.log 2>&1

Restore

# Restaurar em banco de teste (sempre teste antes de prod)
gunzip -c backups/baluarte_2026-06-03_03-00.sql.gz \
  | docker exec -i baluarte-db psql -U baluarte -d baluarte_test
🔴 Teste o restore a cada 30 dias
Backup que não foi testado não é backup. Reserve 15 minutos por mês para restaurar em um banco de teste e confirmar que os dados estão intactos.

Monitoramento & Health Check

Alerta no Telegram quando a API cair, sem custo adicional.

Script de health check

nano /home/deploy/baluarte/healthcheck.sh
#!/bin/bash
WEBHOOK="https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/sendMessage"
CHAT_ID="$TELEGRAM_CHAT_ID"

STATUS=$(curl -o /dev/null -s -w "%{http_code}" \
  "https://api.seudominio.com.br/actuator/health")

if [ "$STATUS" != "200" ]; then
  curl -s "$WEBHOOK" \
    -d "chat_id=$CHAT_ID" \
    -d "text=🚨 Baluarte OFFLINE — /actuator/health retornou $STATUS"
fi
chmod +x /home/deploy/baluarte/healthcheck.sh

crontab -e
# Verifica a cada 5 minutos
*/5 * * * * TELEGRAM_BOT_TOKEN=xxx TELEGRAM_CHAT_ID=yyy \
  /home/deploy/baluarte/healthcheck.sh

Health endpoint no Spring Boot (Actuator)

# application.yml
management:
  endpoints:
    web:
      exposure:
        include: health
  endpoint:
    health:
      show-details: never   # não expõe dados sensíveis

Migração

Checklist de Migração

Railway → VPS Contabo. Siga nessa ordem. Só cancele o Railway depois de tudo validado.

Infraestrutura

  • Contratar Contabo VPS S — Ubuntu 24.04 LTS
  • Ativar pagamento automático no painel Contabo
  • Instalar pacotes: docker, docker-compose-v2, nginx, certbot, ufw, fail2ban, rclone, htop, postgresql-client
  • Configurar firewall UFW (apenas 22, 80, 443)
  • Criar usuário deploy com acesso Docker e sudo
  • Configurar SSH por chave e desabilitar login por senha
  • Tirar primeiro snapshot no painel Contabo

Domínio & SSL

  • Adicionar registro A: api.seudominio.com.br → IP da VPS
  • Aguardar propagação DNS (até 24h — verificar com dig api.seudominio.com.br)
  • Configurar nginx com proxy para 127.0.0.1:8080
  • Emitir SSL: certbot --nginx -d api.seudominio.com.br
  • Testar renovação: certbot renew --dry-run

Banco de dados

  • Exportar banco do Railway: pg_dump -U user dbname > backup.sql
  • Subir container PostgreSQL na VPS: docker compose up -d db
  • Importar banco: psql -U baluarte -d baluarte < backup.sql
  • Validar dados importados

Backend e Docker

  • Criar /home/deploy/baluarte/.env com todas as variáveis reais
  • Subir stack: docker compose up -d
  • Validar health check: curl https://api.seudominio.com.br/actuator/health
  • Verificar logs sem erros: docker compose logs -f

Frontend e integrações

  • Atualizar NEXT_PUBLIC_API_BASE_URL na Vercel
  • Validar fluxo completo: home → login → produto → carrinho → checkout
  • Atualizar webhook Mercado Pago para o novo domínio da API
  • Atualizar webhook SuperFrete
  • Testar pagamento real (valor mínimo)
  • Testar cálculo de frete

CI/CD

  • Adicionar secrets no GitHub: VPS_HOST, VPS_USER, VPS_SSH_KEY
  • Fazer push na main e validar workflow no GitHub Actions
  • Confirmar que container foi atualizado na VPS

Operações

  • Configurar script de backup + crontab (03h00 diário)
  • Configurar rclone com Cloudflare R2 e testar sync
  • Testar restore do backup em banco separado
  • Configurar health check cron com alerta Telegram
  • Agendar snapshot semanal no painel Contabo

Finalização

  • Testar em produção por 48–72h
  • Atualizar sitemap, OG metadata e robots.txt com o novo domínio
  • Cancelar Railway somente depois de tudo validado
🔴 Não cancele o Railway antes de 48h
Deixe os dois ambientes rodando em paralelo. Só cancele após confirmar que pagamentos, fretes, uploads e webhooks funcionam corretamente na nova infra.
